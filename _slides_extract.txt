What  is  Software  Engineering?  
It  is  different  from  Programming  ⇒  It  is  a  systematic  process  for  supporting  software  design,  
development
 
and
 
maintenance
 
activities
 
 All  software  engineers  can  program,  but  not  all  programmers  can  engineer  software.  
Why  Software  Engineering?  
1.  Reduces  complexity:  divides  big  problems  into  several  small  problems  2.  To  minimize  software  cost:  plan  everything   3.  To  decrease  time  4.  Handling  big  projects:  which  requires  lots  of  planning,  management,  resources,  testing  and  
maintenance.
 
 5.  Reliable  software:  working  software  even  if  there  is  a  bug  in  its  given  time  span.  Any  bug  may  cause:  a.  Human  loss  (medical  system)  b.  Economical  or  financial   loss  (Banking  system)  6.  Productivity  :  follow  software  engineering  approach  and  standard  to  increase  productivity  7.  To  meet  the  needs  of  customers.  8.  To  reduce  risk   a.  Project  risk  (Staff  turnover)  b.  Product  risk  (i.e  CASE  tool  underperformance)  c.  Business  risks  (i.e  Competition,  Technology  change))  9.  To  reduce  project  management  overhead

.
 
Goal:

Types  of  Requirements
 
Type  What  it  describes  Examples   
Functional  System  behavior  /  services  
Money  transfer,  email  receipt,  GPS  directions  
“What  the  car  does”  →  starts,  accelerates  
Non-Functional  System  qualities  /  constraints  
Performance,  reliability,  security,  usability  
“How  the  car  performs”  →  fuel  efficiency,  safety  rating,  noise  level  
Domain  Domain-specific  rules  /  regulations  
Compliance  to  standards,  financial  logging  
“Rules  of  the  road”  →  speed  limits,  traffic  signals,  emission  standards.  
 
Key  Activities  in  RE:  
1.  Feasibility  Study:  Technically  and  Financially  (Budget,  resources,  HW,  etc..).  2.  Elicitation:  Gathering  requirements  from  stakeholders,  users,  regulations,  existing  systems,  
and
 
domain
 
experts.
 
Using
 
Interviews
 
(open
 
and
 
closed-
 
predefined
 
questions),
 
questionnaire.
 3.  Analysis:  Refining,  prioritizing,  and  validating  requirements.  4.  Specification:  Documenting  requirements  in  a  clear,  structured  form  (UML,formal  notation,  
tables).
 5.  Validation:  Checking  that  requirements  are  complete,  consistent,  and  feasible.  6.  Management:  Handling  changes  to  requirements  over  time.

Requirement  Validation  Techniques :
 
Validation  Technique  
Explanation  Example  
Formal  /  Informal  Review  
A  manual  analysis  of  requirements  by  stakeholders  to  identify  issues  or  confirm  alignment  with  expectations.  
Formal  Review :  A  meeting  where  the  client,  developers,  and  testers  review  project  requirements  for  a  new  e-commerce  website.  Feedback  includes  changes  to  the  checkout  process.   Informal  Review :  A  discussion  during  development  where  a  designer  notices  the  color  scheme  doesn’t  align  with  brand  guidelines  and  suggests  changes.  
Prototyping  Building  early  versions  of  a  system  or  product  to  visualize  and  test  requirements  before  full  implementation.  
A  clickable  mock-up  of  a  mobile  app’s  user  interface  is  created  to  get  client  feedback  on  navigation  and  functionality  before  final  development.    A  prototype  of  an  online  banking  app  is  built  to  visualize  the  user  flow  of  transferring  money  and  receiving  payments.  Feedback  is  gathered  to  finalize  features  and  UI  elements.  
Test  Cases  Writing  specific  test  scenarios  to  verify  that  each  requirement  can  be  validated  and  tested.  
For  a  requirement  stating  "The  system  should  log  in  users  with  correct  credentials,"  a  test  case  is  created  to  verify  if  entering  the  correct  username  and  password  allows  access.    For  a  requirement  saying  "The  system  must  send  an  email  confirmation  after  each  order  is  placed,"  a  test  case  is  created  to  check  if  the  email  is  sent  after  completing  the  checkout  process.   Examples  of  non-testable requirements:  ●  The  user  interface  must  look  modern  and  attractive.  ●  The  application  must  be  fast.  ●  The  user  interface  must  be  very  responsive.   Examples  of  testable  requirements:  ●  The  system  in  normal  mode  of  operation,  shall  be  able  to  respond  in  6  seconds  or  less

1.  Functional  Requirements  (FRs)  
Describe  what  the  system  should  do  —  the  functions,  tasks,  or  services  provided.  
Examples:  
●  A  banking  app  must  allow  users  to  transfer  money  between  accounts.  
 ●  A  web  store  should  send  an  email  receipt  after  purchase.  
 ●  A  GPS  system  must  provide  turn-by-turn  directions.  
 
Functional  Requirements  (Natural  language)  Problems:  
1.  Ambiguity:  interpreted  in  different  ways  by  developers  and  users.   
 2.  Lack  of  clarity  3.  Requirements  confusion:  Functional  and  non-functional  requirements  tend  to  be  mixed-up.  
2.  Non-Functional  Requirements  (NFRs)  
Definition:  Specify  how  the  system  should  behave ,  rather  than  what  it  does.  Often  called  quality  
attributes
.
 
Examples:  
●  Performance:  The  system  must  handle  10,000  users  simultaneously.  
 ●  Reliability:  The  system  should  have  99.99%  uptime.  
 ●  Security:  All  passwords  must  be  encrypted.  
 ●  Usability:  Users  should  complete  registration  in  less  than  2  minutes.

A  simple  MessageBus  interface   package  com.example.messaging;   import  java.util.*;  import  java.util.function.Consumer;   public  class  MessageBus  {      private  final  Map<String,  List<Consumer<Object>>>  subscribers  =  new  
HashMap<>();
      public  void  subscribe(String  eventName,  Consumer<Object>  handler)  {          subscribers.computeIfAbsent(eventName,  k  ->  new  
ArrayList<>()).add(handler);
     }       public  void  send(String  eventName,  Object  data)  {          List<Consumer<Object>>  handlers  =  subscribers.get(eventName);          if  (handlers  !=  null)  {              for  (Consumer<Object>  h  :  handlers)  {                  h.accept(data);              }          }      }  }     subscribers.computeIfAbsent(eventName,  k  ->  new  ArrayList<>())  means:   “If  eventName  already  has  a  list,  return  it.”   “Otherwise,  create  a  new  ArrayList<Consumer<Object>>,  put  it  in  the  map  under  this  key,  and  
return
 
it.”
   
This  is  a  minimal  event  bus.  send() broadcasts  messages,  subscribe() registers  listeners.

Creating  Maven  Project   
What  is  Maven?  
●  Apache  Maven  is  a  build  automation  and  project  management  tool  for  Java  (and  other  JVM  
languages).
 
 ●  It  helps  you  compile,  package,  test,  and  deploy  your  applications  in  a  standardized  way.  
 ●  It  also  manages  your  project’s  dependencies  (third-party  libraries).  
 
Key  Features  
Feature  What  it  Does  
Build  Automation  Compiles  code,  runs  tests,  packages  JARs/WARs  automatically.  
Dependency  Management  
Downloads  libraries  from  a  central  repository  so  you  don’t  manage  JARs  manually.  
Project  Structure  Encourages  a  standard  directory  layout  (src/main/java ,  src/test/java).  
Plugins  Extend  Maven  to  run  code  analysis,  generate  reports,  deploy  artifacts.  
Convention  over  Configuration  
You  follow  conventions,  and  Maven  does  most  of  the  work  automatically.  
   
298  

===== PAGE 299 =====
  
 
 
How  It  Works  
●  A  Maven  project  is  defined  by  a  POM  file  (pom.xml)  at  the  root.  
 ●  In  pom.xml you  specify:  
 ○  Project  metadata  (name,  version,  packaging)  
 ○  Dependencies  (like  JUnit,  Spring)  
 ○  Plugins  to  run  during  the  build  
 
Maven  then  downloads  the  dependencies  from  the  Maven  Central  Repository  (or  other  repos)  and  
builds
 
your
 
project
 
accordingly.
 
Example:  pom.xml  <project>    <modelVersion>4.0.0</modelVersion>    <groupId>com.example</groupId>    <artifactId>my-app</artifactId>    <version>1.0-SNAPSHOT</version>       <dependencies>      <dependency>        <groupId>org.junit.jupiter</groupId>        <artifactId>junit-jupiter</artifactId>        <version>5.10.0</version>        <scope>test</scope>      </dependency>    </dependencies>  </project>   
 
 
299  

===== PAGE 300 =====
  
 
 
Typical  Commands  mvn  compile          #  Compile  the  source  code  mvn  test             #  Run  tests  mvn  package          #  Package  into  JAR/WAR  mvn  install          #  Install  artifact  into  local  repository  mvn  clean            #  Clean  target  directory    
Why  It’s  Important  
●  No  need  to  manually  download  or  manage  JAR  files.  
 ●  Ensures  builds  are  repeatable  across  machines.  
 ●  Integrates  easily  with  CI/CD  pipelines  and  IDEs  like  IntelliJ  or  Eclipse.
